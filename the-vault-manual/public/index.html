Public Build Workflow (Required)

  1. Define objective, audience, and conversion action for each page before writing content.
  2. Create page-level information architecture and ensure every section has a functional purpose.
  3. Draft with concrete instructions, examples, and user outcomes; avoid abstract product language.
  4. Apply accessibility pass: keyboard navigation, heading order, meaningful labels, and readable contrast.
  5. Run manual quality and split/publisher tests; resolve failures before publish.
  6. Publish to target and verify live response plus render sanity on desktop and mobile.
Gate condition: if structure, utility, or accessibility quality is incomplete, publish is blocked even when tests pass.

Professional Quality Expectations

100% Sections must serve a concrete user task
0 Placeholder tokens permitted at publish time
AA+ Accessible interaction and reading flow target
1:1 Design intent must match content utility
Design quality is not decorative only: aesthetics are treated as comprehension infrastructure for front-facing outputs.
Narrative integrity check: public language must still reflect core runtime truth, including Vault_Brain orchestration and Health-informed prioritization where relevant.
Vault-wide execution doctrine: process writeups can never substitute for artifact delivery; artifact polish can never substitute for governance/process objectives when both are requested.
