A. Executive Overview

The Vault is a local-first, personally owned AI operating system. It coordinates a 20-persona agent system, a multi-layer orchestration brain, a governed tool registry, and a rich set of domain-specific capabilities spanning memory, scheduling, health, finances, shopping, social communication, and creative work.

The system runs locally in Python on Windows and calls large language models (LLMs) via OpenRouter as the primary gateway. No data is sent to any cloud service except through explicitly configured API calls.

Design principles

  • The Vault is authoritative. KAOS is the sandbox — it is never the source of truth.
  • Secrets stay in .env. Nothing sensitive is committed to git.
  • Every write-capable or destructive action is gated behind human approval.
  • All agent-callable functions use the api_ naming convention.
  • Security gates SR-001, SR-002, and SR-003 are all resolved and enforced in production.

C. Quick Start — Operator Guide

Prerequisites

  • Python 3.11+ installed
  • Git configured with push access to FerretFatale/KAOS
  • A valid .env file at the repo root with at minimum OPENROUTER_API_KEY set

Step 1 — Install dependencies

cd C:\Users\LittleChickpea\OneDrive\Documents\The_Vault
pip install -r requirements.txt

Step 2 — Run the test suite

python -m pytest Copilot_Tests/ -q --tb=short

Expected: 800+ tests passing, 0 failures.

Step 3 — Check system state

# Persona count
python -c "import json; m=json.load(open('persona_manifest.json')); print(len(m), 'personas:', list(m.keys()))"

Step 4 — Start the system

python boot_kaos.py

What Copilot agents must do at task start

  1. Read AGENTS.md
  2. Read docs/copilot-context/system-context.md and authority-map.md
  3. Run git log --oneline -10 and git status --short
  4. Read KAOS OS ACTIVE BACKLOG.txt for unchecked [ ] tasks
  5. Read Human_Input.txt for active blockers
  6. Identify the single next unchecked task and verify it is not already done
  7. Implement, test, stage, and commit only that task

Common mistakes to avoid

  • Do not stage Health/*.json, Recovery/*.json, or other runtime churn files
  • Do not edit .kaos_mission.json by hand during an active mission
  • Do not add secrets or API keys to committed files — use .env only
  • Do not run git add -A without reviewing what will be staged

E. Vault_Brain

Vault_Brain (vault_brain.py) is the system coordinator operating at temperature 0.0. It routes tasks to appropriate personas, tracks human action entries, manages proposals, and controls operating modes.

Key responsibilities

  • Task routing to appropriate personas
  • Human-action entry management (Human_Input.txt)
  • Proposal queue management
  • Backlog phase coordination
  • Deploy guard enforcement (SR-002)
  • Context budget management (E2: context-purge wiring)

Thinking quality improvements: E1 (success-criteria injection), E2 (context-purge wiring; memory notes truncated to 500 tokens).

F. TaskMaster_Brain

TaskMaster_Brain (TaskMaster_Brain.py) is the mission orchestrator and LLM planner. It generates step-by-step execution plans, dispatches steps to personas, and enforces planning discipline.

Key rules

  • Rule 5 — PRESERVED STEPS: Steps marked complete must not be re-planned.
  • Core skill disambiguation: Every persona has a core_skill field. Missing core_skill defaults to "Data Processing".
  • M4/T4 risk fields: Mission and task records carry risk metadata.

G. Consensus_Engine

Consensus_Engine (Consensus_Engine/) is the governance auditor and decision scorer. It validates phase transitions, scores consensus outcomes, and audits decisions. It is read-only — no side effects.

api_transition_phase is explicitly excluded from Consensus_Engine by design — it belongs to Vault_Brain.

Capabilities (34 tools): voting and consensus scoring, governance logging, phase read/validate operations, state read-only access and analytics.

H. Contextual_Thinking

Contextual_Thinking (Contextual_Thinking/) is the thinking quality layer. It manages contextual rules, persona-level thinking preferences, and context-window budget strategies.

Capabilities (12 tools): context selection, persona-aware prompt shaping, reasoning scaffolds.

Publisher CLI

python Toolkit/manual_publisher.py --prepare --target web-sandbox-1

# Dry-run publish
python Toolkit/manual_publisher.py --publish --target web-sandbox-1 --dry-run

# Live copy + push
python Toolkit/manual_publisher.py --publish --target web-sandbox-1 --approved

# Publish variant only
python Toolkit/manual_publisher.py --publish --target web-sandbox-1 --variant developer --approved

# Verify hosted output
python Toolkit/manual_publisher.py --verify --target web-sandbox-1

Publish path

Files are published under /the-vault-manual/ in the target repo. Subdirectories (developer/, public/) are preserved via relative_to(_MANUAL_DIR) copy logic.

Change ledger

Each manual rebuild appends a record to docs/manual/manual_change_log.jsonl. The ledger tracks version, source hash, sections changed, and timestamp. Use it to audit what changed between builds and why a new publish was needed.

Safety gates (always enforced)

  • dry_run=True is the default — no files copied unless explicitly set to False
  • publish_approved=False prevents any git push
  • The_Vault and KAOS are refused as targets regardless of arguments
  • Dirty target repos are refused unless allow_dirty_target=True
  • Forbidden files (.env, Human_Input*.txt, Health/*.json, etc.) are never copied

BB. Troubleshooting and Recovery

Manual appears stale

python Toolkit/manual_updater.py --staleness-report

Persona count mismatch

python -c "import json; m=json.load(open('persona_manifest.json')); print(len(m), list(m.keys()))"

Test failures after a Copilot pass

python -m pytest Copilot_Tests/ -q --tb=short

Failed Copilot run recovery

  1. Run git status --short
  2. Check git diff HEAD for partial changes
  3. Do NOT stage runtime churn files
  4. Check Human_Input.txt for newly appended blockers
  5. Use git restore --staged <file> to unstage accidents
