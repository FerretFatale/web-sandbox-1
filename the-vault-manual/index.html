Overview

The Vault is a local-first, personally owned AI operating system. It coordinates a 20-persona agent system, a multi-layer orchestration brain, a governed tool registry, and a rich set of domain-specific capabilities across memory, scheduling, health, finances, shopping, communication, and creative work.

The system runs locally (Python) on Windows and calls LLMs via OpenRouter as the primary gateway. No data is sent to any cloud service except through the explicitly configured API calls. No background sync or subscription model is in use.

Design principles

  • The Vault is authoritative. KAOS is sandbox-only and never overrides production.
  • Secrets stay in .env. Nothing sensitive is committed.
  • Every write-capable or destructive action is gated behind human approval.
  • All agent-callable functions use the api_ naming convention.
  • SR-001, SR-002, and SR-003 security gates are all RESOLVED and enforced.

Architecture

The system has four main layers:

  1. Brain layer — orchestration, planning, consensus, and thinking quality (Vault_Brain, TaskMaster_Brain, Consensus_Engine, Contextual_Thinking)
  2. Persona layer — 20 specialist personas each with curated tool grants and delegated scope
  3. Tool registry layer — custom_tools_registry.py + persona_manifest.json + Skills_Index.json
  4. Domain layer — memory, schedule, health, finances, shopping, social, email, business, security

Quick Start — Operator Guide

This section walks through what an operator or developer needs to do to get started with The Vault from a fresh checkout, or to re-orient after a gap. It covers environment setup, first run, and where to look for current system state.

ℹ️ If you are a Copilot agent, read AGENTS.md and the context files in docs/copilot-context/ before starting any task. Do not start implementation before reading current repo state — verify before acting.

Prerequisites

  • Python 3.11+ installed
  • Git configured with push access to FerretFatale/KAOS (The_Vault repo)
  • A valid .env file at the repo root with at minimum OPENROUTER_API_KEY set

Step 1 — Install dependencies

cd C:\Users\LittleChickpea\OneDrive\Documents\The_Vault
pip install -r requirements.txt

Step 2 — Run the test suite

python -m pytest Copilot_Tests/ -q --tb=short

Expected: 800+ tests passing, 0 failures. If tests fail before you have made any changes, check git log --oneline -5 and compare against the last known-good commit.

Step 3 — Check system state

# Persona count
python -c "import json; m=json.load(open('persona_manifest.json')); print(len(m), 'personas')"

# Open KAOS OS ACTIVE BACKLOG.txt — look for [ ] items (not-started tasks)
# Open Human_Input.txt — look for TASK: blocks (active human blockers)

Step 4 — Start the system

python boot_kaos.py

This starts the main boot sequence. The system initializes personas, checks credentials from .env, and reports readiness. No external network calls happen at boot unless an LLM call is triggered.

What Copilot agents must do at task start

  1. Read AGENTS.md
  2. Read docs/copilot-context/system-context.md and authority-map.md
  3. Run git log --oneline -10 and git status --short
  4. Read KAOS OS ACTIVE BACKLOG.txt for unchecked [ ] tasks
  5. Read Human_Input.txt for active blockers
  6. Identify the single next unchecked task and verify it is not already done in current files
  7. Implement, test, stage, and commit only that one task

Common mistakes to avoid

  • Do not stage Health/*.json, Recovery/*.json, Toolkit/.backlog_pass_log.json, or other runtime churn files (see Runtime Churn)
  • Do not edit .kaos_mission.json by hand during an active mission
  • Do not add secrets or API keys to committed files — .env only
  • Do not run git add -A without reviewing what will be staged
  • Do not mark a task [x] unless code is committed and tests pass
  • Do not trust backlog markers without verifying actual repo state

Vault_Brain routes tasks, tracks human actions, manages proposals, and controls operating modes. It does not perform domain execution (writing code, creating content, running schedules).

  • Human-action entry management (Human_Input.txt)

  • Backlog phase coordination
  • Deploy guard enforcement (SR-002)

TaskMaster_Brain

TaskMaster_Brain.py — mission orchestrator and LLM planner.

TaskMaster_Brain generates step-by-step execution plans, dispatches to personas, and enforces planning discipline. It is the entry point for multi-step agent missions.

Planner rules (key)

  • Rule 5 — PRESERVED STEPS: Steps marked complete in mission state must not be re-planned. This prevents wasteful re-execution of already-completed work.
  • Core skill disambiguation: Every persona has a core_skill field. TaskMaster_Brain uses it to route ambiguous requests to the right specialist.

Consensus_Engine validates phase transitions, scores consensus outcomes, and audits decisions. It does not execute missions, write domain content, or perform side-effect actions. It does not own api_transition_phase (destructive mutation — excluded by design).

Backlog System

The backlog is the authoritative task-order source for Copilot execution passes.

Task status markers

MarkerMeaning
[ ]Not started — agent will pick this up next
[~]Partially complete, blocked, or awaiting human input
[x]Complete and verified — do not re-execute

Backlog rules

  • Process one unchecked item at a time in file order
  • Never reorder items; only append new ones or mark complete
  • Mark [x] only after code is committed and tests pass
  • If blocked: mark [~] and append to Human_Input.txt
  • Do not put completed work back in active backlog
  • Verify current repository state before trusting backlog status

Duplicate / stale cleanup expectations

  • The template section of the backlog contains [x] TASK: <clear task title> — this is a format sentinel, not a real task. The backlog lint test recognises this sentinel and skips it correctly.
  • When a task is fully archived, move it to KAOS OS BACKLOG ARCHIVE.txt

Manual Publishing Workflow (Backend)

The_Vault includes a backend publishing system so the operator never manually copies manual files into website repos. The workflow is:

  1. The_Vault rebuilds the manual from docs/manual/manual_source.md + section edits
  2. Changes are recorded in docs/manual/manual_change_log.jsonl
  3. Toolkit/manual_publisher.py copies approved files to the target repo
  4. Publisher validates no secrets or forbidden files are included
  5. Publisher commits and pushes the target repo when publish_approved=True
  6. Publisher verifies the hosted URL after push

Safety requirements enforced by the publisher

  • Default dry_run=True — no files are copied without explicit override
  • No push unless publish_approved=True is explicitly set
  • Target path check — refuses if path contains The_Vault or KAOS
  • Dirty-target check — refuses to publish to a dirty repo unless allow_dirty_target=True
  • Forbidden-file filter — never copies .env, Human_Input*.txt, Health/*.json, Recovery/*.json, or any runtime personal data
  • Only copies from safe_copy_patterns defined in manual_publish_targets.json

Config file

Target paths and safety rules are stored in docs/manual/manual_publish_targets.json. When a sandbox repo is cloned, update local_path in that file and re-run the publisher.

Troubleshooting

Manual appears stale

  1. Run python Toolkit/manual_updater.py --staleness-report
  2. Check docs/manual/manual_change_log.jsonl for pending entries
  3. Run morning batch dry-run to see what would change
  4. If persona count is wrong, check persona_manifest.json (count top-level keys)
  5. If test count is wrong, run python -m pytest Copilot_Tests/ --collect-only -q | tail -5

Persona count mismatch

Expected: 20 personas in persona_manifest.json.

python -c "import json; m=json.load(open('persona_manifest.json')); print(len(m), list(m.keys()))"

If count differs, the manifest was edited without updating the manual. Check git log for recent manifest changes and update accordingly.

Test failures after a Copilot pass

  1. Run python -m pytest Copilot_Tests/ -q --tb=short to get the failure list
  2. Common causes: stale string assertions (test checks for a specific count/status that changed), import path changes, generated file not regenerated
  3. For shopping assistant failures: check if ALDI HTML structure changed
  4. For thinking contract failures: check if vault_brain.py E1/E2 wiring changed

Failed Copilot run recovery

  1. Run git status --short to see what changed
  2. Check git stash or git diff HEAD for partial changes
  3. Do NOT stage runtime churn: Health/*.json, Recovery/*.json, Toolkit/.backlog_pass_log.json, Toolkit/.hf_preferences.json, Images/Morning_Briefings/*, Onboarding/scan_log.json, Schedule/reminders.json

What if churn files appear in a PR?

  • Do not merge. Unstage the churn files and amend the commit or create a fresh commit without them.

How to Update This Manual

The manual has two source files. Always edit the prose source first, then propagate to HTML.

Source files

FileRole
docs/manual/manual_source.mdCanonical prose source — edit this for all content changes
docs/manual/index.htmlBuild output — rendered/updated from manual_source.md

Standard update procedure

  1. Edit docs/manual/manual_source.md with the updated prose
  2. Apply the same change to the relevant section in docs/manual/index.html
  3. Update docs/manual/manual_meta.json — set last_updated for the affected section(s)
  4. Run staleness check: python Toolkit/manual_updater.py --staleness-report
  5. Run manual tests: python -m pytest Copilot_Tests/test_manual_updater.py -q
  6. Commit with: docs(manual): update [section name]

When to log a deferred update instead

If a capability change is made but the manual cannot be updated in the same session (too large, needs design review, waiting on blocked items), log the change for the next batch:

from Toolkit.manual_updater import api_log_manual_change_needed
api_log_manual_change_needed(
    source_file="Toolkit/new_feature.py",
    subsystem="new-feature",
    affected_section="blocked",
    reason="new-feature implemented but manual section not yet created"
)

When to increment the version

  • v3 → v4 (major) — architecture redesign, new brain tier added, 5+ new sections, major persona restructure
  • v3 → v3.1 (minor) — new sections added, significant prose improvements, new personas or tools documented
  • No version bump needed — typo fixes, stale status tags updated, minor wording improvements

Update the version in <title>, the header meta-card, and docs/manual/manual_meta.json when bumping.

Do not

  • Do not edit index.html without updating manual_source.md — they must stay in sync
  • Do not invent capabilities — only document features that exist, are explicitly blocked, or are proposal-first
  • Do not add inline <script> or external JavaScript to this file — it is a static docs-only output
  • Do not remove sections without archiving the content in manual_source.md
  • Do not use this file as a planning scratch-pad — write to session plan files instead
