# External Skills and Assimilation

When The_Vault uses an external skill/tool/plugin to fill a gap, follow a deterministic closure loop.

## Assimilation loop

1. Capture gap and reason external capability was needed.
2. Capture exact external surface used.
3. Record outcome and reproducibility.
4. Decide one path:
- Adopt now
- Scaffold now
- Defer
5. Enshrine decision and update governance references.

## Decision matrix (quick)

1. **Adopt now**
	- Use when capability is stable, repeatable, and clearly valuable.
2. **Scaffold now**
	- Use when partial integration is useful but full adoption is premature.
3. **Defer**
	- Use when risk, complexity, or low value does not justify immediate work.

## Beginner-friendly assimilation workflow

1. Describe the exact gap in plain language.
2. Show current internal capability first.
3. Use external capability only for the missing part.
4. Validate result with deterministic evidence.
5. Record what was learned and whether to adopt/scaffold/defer.

## Example triggers

1. "I need a third-party API capability we do not currently implement."
2. "I need a specialized formatting/export path missing from current tools."
3. "I need a one-time external lookup to complete a lane."

## Ownership

1. `TaskMaster_Brain`: ensure assimilation step appears in planning where relevant.
2. `Vault_Architect`: final assimilation closure owner.
3. `Business_Ops`: provide explicit recommendation for project-builder lanes.

## Why this matters

This prevents repeated dependence on ad-hoc external capability usage and turns wins into stable internal process intelligence.

It also keeps user experience predictable: fewer surprises, more repeatable outcomes.
